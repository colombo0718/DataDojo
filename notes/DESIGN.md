## 核心哲學：讓不可見的變化變得可見

教學平台的價值不在於「提供功能」，而在於**讓學生在操作的當下產生真正的理解**。
一個功能點下去，要能讓人發出「喔，原來是這樣」的感嘆。

---

## 功能設計五問

設計任何互動功能時，依序回答這五個問題。答不出來就不該做這個功能。

**1. 這個操作在教什麼概念？**
說得清楚具體的概念，不是模糊的「讓學生理解 XX」。

**2. 「喔原來是這樣」的瞬間是什麼？**
要能描述那個頓悟的具體畫面。
不是「學生會理解正規化」，而是「學生看到 x 軸從 0~500 縮進 0~1，形狀沒變」。

**3. 那個瞬間是由操作本身直接觸發的嗎？**
如果學生要去別的地方找證據，設計就失敗了。
反例：Weka 按下 Apply，要自己去圖表軸上找數字才確認有沒有做到正規化。

**4. Before / After 同時可見嗎？**
任何資料或狀態的變換，對比必須是主動展示的，不是讓學生自己比對。

**5. 不看任何說明文字，視覺能自己說話嗎？**
如果需要一段解釋才能看懂這張圖，圖就還沒設計好。

## Pipeline 積木（Blockly as Macro）

DataDojo 的 Blockly 角色不是「寫程式」，而是**記錄與編輯操作歷程的巨集系統**。

每一個資料處理動作（正規化、移除離群值、選特徵⋯⋯）都對應一個積木。
學生的操作歷程 = 一排積木序列，可視、可編輯、可重排、可 do/undo。

### 積木分類即 Pipeline 分類

積木的抽屜分類不用傳統程式語言的「邏輯、字串、數學」，而是：

```
資料描述  |  數據清理  |  資料前處理  |  訓練/測試  |  結果解讀
```

與 pipeline 地圖、右欄流程指示器使用同一套語言，三個地方強化同一個心智模型。

### 這個設計同時解決的問題

| 問題 | 解法 |
|------|------|
| 操作 log 只能讀，不能改 | 積木序列可互動，可拖動重排、刪除單步 |
| 學生不知道操作屬於哪個階段 | 積木分類即階段，打開抽屜就知道 |
| 評量只能看結果 | 老師看學生的積木序列 = 看思考過程 |
| 學生不知道順序是否有影響 | 對調積木，立刻看到結果差異 |
| DD 到 Kaggle 的橋 | 每個積木對應一段 Python/pandas 程式碼，並排顯示 |

### 與 RR Blockly 的本質差異

RR 的 Blockly：**創造邏輯**（寫 policy 腳本，決定 Agent 如何行動）
DD 的 Blockly：**紀錄與組織行為**（記錄對資料做了什麼）
