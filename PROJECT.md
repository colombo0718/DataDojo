# PROJECT.md — DataDojo 專案快速指引

## 這個專案是什麼

**DataDojo (DD)** 是 LeafLune 旗下的互動式機器學習教育平台，讓使用者在瀏覽器中體驗監督學習與非監督學習。
與 Rein Room (RR) 並列為 LeafLune 教育工具體系兩大支柱：RR 主攻強化學習，DD 主攻傳統 ML。

**一句話定位：DataDojo 讓老師看得見學生的思考過程，讓學生看得見資料的變化過程。**

### 解決的痛點：理論與實作之間的斷層

現有 ML 學習路徑存在一個明顯斷層：

```
理論影片 / 教材（生動圖解、直觀易懂）
        ↓
        ← 斷層：認知跳躍太大 →
        ↓
實際操作（Python 程式碼 / Weka 冰冷介面）
```

學生看完影片懂了概念，但一碰到真實資料就要面對程式語法或令人困惑的工具介面，學習動力在這裡斷掉。

**DataDojo 填補這個斷層**：看完理論，立刻在親民的視覺介面上操作真實資料，親眼看到算法行為，不需要先學會寫程式。這也是 DD 搭配勁園教材的核心特色——教材講理論，DD 是理論的「立即驗證台」，課堂上講完就能開。

### 與競品的定位切分

| | Weka | Kaggle | DataDojo |
|---|---|---|---|
| 核心動詞 | 執行 ML | 競爭 / 實戰 | 理解 ML |
| 假設的用戶 | 已知道自己要做什麼 | 已會寫程式 | 剛接觸，不知道「為什麼」 |
| 成功定義 | 跑出結果 | 名次、分數 | 「啊，我懂了」 |

Weka 是專業工具，沒有教學基因。DD 和 Kaggle 都有「理解優先」的教學基因，差異是語言：DD 用視覺語言，Kaggle 用程式語言。DD 是入口，學生帶著直覺從 DD 銜接到 Kaggle。

### 受眾深度分層（同一工具，不同出口）

```
中小學        → 資料描述即終點（平均數、標準差、分布圖）
技職 / 高中   → 資料描述 + 清理（資料品質、異常值、關聯性）
大學          → 走完整個 pipeline（模型訓練、評估、結果解讀）
```

不用模式選擇，介面設計自然讓用戶決定走多深（漸進揭露）。

### 兩個核心使用場景

**學習時** — 好看、好操作、有引導感，調參數立刻看到效果變化，建立算法直覺。

**考試 / 評量時** — UI 操作 log + 最終數據輸出，讓老師看到學生做了什麼（流程），不只是得到什麼結果。搭配 MOSME 教室系統，學習歷程可視化。

## 核心功能規劃

### 算法層（ml.js）
| 類別 | 算法 |
|------|------|
| 監督 - 分類 | k-NN、Decision Tree、Naive Bayes、Random Forest、SVM |
| 監督 - 迴歸 | 線性迴歸、多項式迴歸 |

- 決策邊界圖
- 混淆矩陣
- ROC 曲線
- 分群散點圖
- PCA / UMAP 投影圖

### 資料層
- CSV 上傳與解析
- 內建示範資料集（Iris、MNIST 子集等）
- 缺失值處理、特徵正規化

## 與 RR 的連動

| 方向 | 說明 |
|------|------|
| RR → DD | RR 的 SAR 軌跡、Q-table 可匯入 DD 做 UMAP 分群、PCA 分析（找不同打法流派） |
| DD → RR | DD 訓練出的分類器（state → action）可匯入 RR 當作初始 policy（Behavioral Cloning） |

共通格式：`(state_vec, action_vec, reward, next_state_vec, done)`

## 介面佈局

RR 的左右分欄架構沿用至 DD：

```
左欄：處理對象          右欄：處理工具
─────────────────    ─────────────────────
RR → 遊戲畫面          訓練控制、參數調整、圖表
DD → 資料表格          Pipeline 階段、算法選擇、視覺化
```

DD 的左欄是原生資料表格，不需要 iframe，實作比 RR 簡單。

## DD 作為 RR 互動教學系統的驗證場

DD 處理的是靜態資料，具備**確定性**：相同的操作流程，保證得到相同的結果。

這讓 DD 成為打磨「可互動教學平台」概念的最佳場域：
- 教師可以精確展示操作流程，學生跟著做結果可驗證
- 操作 log 的評量機制在 DD 裡可靠——結果不同，就是某一步做錯了
- 教室系統（班級碼、學習歷程、評量）先在 DD 建立與驗證，再移植回 RR

RR 的 RL 訓練有隨機性，相同操作不保證相同結果，評量難度更高。DD 先行，累積經驗後再解決 RR 的評量問題。

## UX 設計原則

- **漸進揭露**：不鎖關卡，介面自然引導深度，功能永遠在那裡但只在需要時出現
- **地圖而非走廊**：流程指示器是「你在地圖上的位置」，學生可回頭、可跳，但隨時知道自己在哪
- **理解優先**：核心迴圈是「調參數 → 立刻看到效果 → 產生直覺」，不是「跑出數字」
- **資料品質可見**：讓學生親眼看到資料清理前後對模型的影響，這是 Weka / Kaggle 都沒做好的
- **故事思維**：每個算法有它要回答的問題，學生跟著問題走，不是跟著 toolbar 走
- **操作 log**：記錄學生的每一步操作，供老師評量與 MOSME 成績管理銜接
